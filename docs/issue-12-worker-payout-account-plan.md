# Issue #12 振込先情報登録 — 改訂実装プラン

## 1. 目的

稼働者本人が振込先を登録・更新でき、管理者が支払い実務時に最新情報を確認できるようにする。

振込先は機密情報として扱い、表示画面、サーバーレスポンス、DB 保存、ログ出力を必要最小限にする。

関連 Issue: [#12 振込先情報をシステムに登録できるようにする](https://github.com/techguide-jp/techguide-jp/issues/12)

## 2. 設計方針

- 振込先は `worker_profiles` とは別テーブルに保存する。
- 振込先詳細はアプリケーション層で暗号化して保存する。
- 全項目を表示する画面は `/workers/[login]` に集約する。
- 精算画面には登録状態、更新日時、プロフィールへのリンクだけ表示する。
- 本人だけが登録・更新できる。
- 管理者は閲覧のみ可能とする。
- 未登録でも月次申請・承認はブロックせず、警告のみ表示する。
- 部分保存は許可せず、レコードが存在すれば「登録済み」と判定する。
- 削除・登録解除は今回の対象外とする。
- 振込先更新は精算スナップショットの差分判定に含めない。

## 3. スコープ

### 3.1 対象

- 振込先情報の新規登録・更新
- 本人および管理者による権限に応じた閲覧
- 更新者・更新日時の記録
- 月次精算画面での登録状態表示
- 保存データの暗号化
- 必要な migration、テスト、ドキュメント更新

### 3.2 対象外

- 銀行振込の実行や決済代行サービス連携
- 口座番号の自動照合や銀行 API 連携
- マイナンバーなどの本人確認・税務情報
- 確認済み・要修正などのステータス管理
- 振込先情報の更新履歴・監査ログ
- 支払い状況ステータスとの連携
- 支払い通知書への振込先表示
- 振込先情報の削除・登録解除
- 暗号鍵ローテーションの自動化

## 4. 権限

| 操作         |     本人 | 管理者 | その他 |
| ------------ | -------: | -----: | -----: |
| 登録・更新   |        ○ |      × |      × |
| 全項目閲覧   |        ○ |      ○ |      × |
| 登録状態閲覧 | 自分のみ |   全員 |      × |

未認可アクセスは既存仕様に合わせて `/work` へ `303` リダイレクトする。

レスポンス制御は以下とする。

- 本人には自分の全項目を返す。
- 管理者には対象者の全項目を返す。
- その他の利用者には振込先フィールド自体を返さない。
- 月次一覧・月次詳細では、管理者にも登録状態と更新日時だけ返す。

## 5. 画面・導線

### 5.1 `/workers/[login]`

本人向け:

- 振込先登録・更新フォーム
- 登録済み・未登録表示
- 最終更新日時
- 保存完了・入力エラーメッセージ

管理者向け:

- 読み取り専用の振込先情報
- 更新者、最終更新日時
- 編集操作は表示しない

その他の利用者:

- ページを表示せず `/work` へリダイレクトする。

### 5.2 `/settlements/[month]/[assignee]`

本人・管理者共通で以下を表示する。

- 振込先の登録状態
- 最終更新日時
- 未登録時の警告
- `/workers/[login]` へのリンク

この画面の load レスポンスには、口座番号、口座名義、補足メモなどの詳細を含めない。

### 5.3 `/settlements/[month]`

管理者向け一覧へ以下を追加する。

- 振込先列
- 登録済み・未登録表示
- 最終更新日時
- 対象プロフィールへのリンク

## 6. 入力項目・バリデーション

| 項目       | 必須 | 仕様                           |
| ---------- | ---: | ------------------------------ |
| 金融機関名 |    ○ | `trim`、1〜100文字             |
| 支店名     |    ○ | `trim`、1〜100文字             |
| 口座種別   |    ○ | 普通・当座・貯蓄               |
| 口座番号   |    ○ | 全角数字を半角化し、数字7桁    |
| 口座名義   |    ○ | 全角カナへ正規化し、1〜100文字 |
| 補足メモ   |      | `trim`、最大2000文字           |

- 口座番号は先頭ゼロを保持するため文字列として扱う。
- ゆうちょ銀行の記号番号は直接扱わず、他行からの振込用に変換済みの支店名・7桁口座番号を入力する。
- 口座名義では、全角カナ、長音、空白、業務上必要な一般的記号を許可する。
- バリデーションエラーには入力値を含めず、項目単位の一般的なメッセージだけを返す。

## 7. DB 設計

### 7.1 `worker_payout_accounts`

```text
login                    PK / FK → worker_profiles.login
encrypted_payload        text NOT NULL
encryption_key_version   integer NOT NULL DEFAULT 1
updated_by               text NOT NULL
version                  integer NOT NULL DEFAULT 1
created_at               timestamptz NOT NULL DEFAULT now()
updated_at               timestamptz NOT NULL DEFAULT now()
```

暗号化対象の payload:

```ts
type WorkerPayoutAccountPayload = {
  bankName: string;
  branchName: string;
  accountType: "ordinary" | "checking" | "savings";
  accountNumber: string;
  accountHolderName: string;
  note: string;
};
```

DB 制約・運用方針:

- `login` は `worker_profiles.login` を参照し、`ON DELETE CASCADE` とする。
- 登録済み判定はレコードの存在で行う。
- 更新時は `version` を使い、多重タブなどによる古い内容の上書きを検知する。
- 検索や一覧取得には暗号化 payload を使用しない。
- 一覧取得では `login` と `updatedAt` だけを取得する。

## 8. 暗号化方針

- AES-256-GCM を使用する。
- 環境変数 `PAYOUT_ACCOUNT_ENCRYPTION_KEY` に 32 byte の鍵を Base64 形式で設定する。
- nonce、認証タグ、暗号文を一つの保存形式へエンコードする。
- 復号は本人または管理者向け詳細取得時だけ行う。
- 復号失敗時に暗号文や入力値をエラーへ含めない。
- 暗号鍵をログへ出力しない。
- `.env.example` には設定方法だけを記載し、実際の鍵を含めない。
- `encryption_key_version` を保持し、将来の鍵ローテーションへ対応可能にする。

## 9. レイヤー構成

```text
src/lib/server/db/schema.ts

src/lib/server/payoutAccounts/
  payoutAccountCrypto.ts
  payoutAccountRepository.ts
  payoutAccountService.ts
  payoutAccountTypes.ts

src/routes/workers/[login]/
  +page.server.ts
  +page.svelte

src/routes/settlements/[month]/[assignee]/
  +page.server.ts
  +page.svelte

src/routes/settlements/[month]/
  +page.server.ts
  +page.svelte
```

主要サービス:

```ts
loadPayoutAccountForViewer(targetLogin, viewer);
updateOwnPayoutAccount(formData, actorLogin, targetLogin);
getPayoutAccountStatus(login);
listPayoutAccountStatuses(logins);
```

一覧用の型:

```ts
type WorkerPayoutAccountStatus = {
  login: string;
  registered: boolean;
  updatedAt: Date | null;
};
```

振込先状態は既存の `SettlementSummary` へ追加せず、route load または表示用サービスで別データとして結合する。これにより、振込先更新が精算スナップショットの差分判定へ影響することを防ぐ。

## 10. セキュリティ要件

- 口座情報を監査ログへ記録しない。
- `console`、エラーログ、監視サービスへ FormData や復号済みデータを渡さない。
- バリデーションエラーへ入力値を含めない。
- 月次一覧・月次詳細のレスポンスへ暗号化 payload や復号済み詳細を含めない。
- 未認可利用者向けレスポンスには振込先プロパティ自体を含めない。
- E2E では明確なダミー口座番号だけを使用する。
- 本番 DB 参照手順で暗号化 payload を不用意に出力しない。
- 口座番号入力へ `inputmode="numeric"` を設定する。
- 保存・復号処理で発生した例外は一般化したメッセージへ変換する。
